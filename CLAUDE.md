# Инструкции для агента

## Язык
Всегда отвечай на русском языке. Код, имена переменных, команды, конфиги — на английском, как и положено.

## Контекст проекта
Dream Diary Bot — Telegram-бот для записи и поиска снов на Python (`python-telegram-bot`), хранение в SQLite. Структура: `bot.py`, `db.py` + вспомогательные файлы. Работа ведётся в `.venv`. Поиск по Cyrillic-тексту регистронезависим через `str.casefold()` на стороне Python (SQLite `LIKE` для кириллицы не годится). Есть изоляция данных по `user_id`.

Базовая функциональность уже реализована: добавление, просмотр списка, поиск, удаление записей. У бота два параллельных интерфейса к одной и той же логике — инлайн-кнопки (главное меню) и текстовые команды (`/add`, `/list`, `/search`, `/delete` и т.д.) как резервный способ. Сейчас приоритет — не новые фичи, а надёжность: отсутствие ошибок в существующих командах и аккуратное, проверенное добавление нового функционала.

## Рабочий процесс (обязательно соблюдать)

### Шаг 1 — Анализ
При начале работы с задачей:
- Изучи структуру файлов и зависимости, затронутые задачей
- Прочитай относящиеся к задаче части `bot.py`, `db.py` и README
- Сформулируй краткое описание того, что сейчас происходит в этом участке кода (3–5 предложений)
- Если задача — это баг: воспроизведи логику мысленно (или предложи шаги для воспроизведения) прежде чем предполагать причину

### Шаг 2 — Предложение вариантов
Перед любыми изменениями:
- Предложи 2–3 варианта решения с плюсами и минусами каждого
- Укажи, какие файлы и какие команды бота затронет каждый вариант
- Отдельно отметь риски: может ли вариант сломать другую команду, потерять данные пользователя, нарушить изоляцию по `user_id`
- ЖДИ подтверждения от пользователя

### Шаг 3 — Действие
Только после явного подтверждения ("да", "делай", "вариант 2"):
- Вноси изменения итеративно, небольшими шагами — один логический кусок за раз
- После каждого шага кратко сообщай что сделано

### Шаг 4 — Проверка функциональности
После внесения изменений, до отчёта о завершении:
- Запусти существующие тесты, если они есть; если тестов нет — предложи добавить хотя бы минимальный тест для изменённой логики
- Проверь не только саму изменённую команду, но и соседние команды, которые могли задеть изменения (например: правка в `db.py` — проверь add, list, search, delete все разом)
- Если изменение затрагивает функционал, доступный и через кнопку, и через текстовую команду — проверь оба пути отдельно, это разные точки входа в одну логику и баг может проявиться только в одной из них
- Явно проверь edge-cases: пустая база, кириллица в поиске, несколько пользователей одновременно, пустой ввод от пользователя, ручной override даты через `dream_date` наряду с автоматической датой
- Если автоматической проверки недостаточно — опиши пользователю, какие шаги вручную стоит проверить в самом Telegram

### Шаг 5 — Обновление документации
Если изменения затрагивают команды бота, кнопки меню, структуру БД, способ запуска/деплоя, зависимости или поведение, видимое пользователю:
- Предложи конкретную правку в README — включая, при необходимости, таблицу команд, раздел про кнопки, SQL-схему таблицы `dreams` и секцию деплоя (systemd/Docker), если меняется запуск или зависимости
- ЖДИ подтверждения перед правкой README, как и для кода

## Поиск и исправление ошибок

- При расследовании бага сначала локализуй его: какая функция, какая команда, при каких входных данных
- Не исправляй "на глаз" — сначала сформулируй гипотезу о причине и проверь её (логами, чтением кода, тестовым вызовом), и только потом предлагай фикс
- Если баг системный (например, общая проблема обработки кодировки или общая проблема работы с БД) — предупреди, что фикс в одном месте может не покрыть остальные похожие случаи, и укажи где ещё могут быть те же проблемы
- После фикса — обязательно прогон по Шагу 4 (проверка), а не просто "должно работать"

## Запреты
- Не изменяй файлы без явного подтверждения
- Не удаляй код без предупреждения
- Не устанавливай зависимости без спроса
- Не меняй схему БД и не пиши миграции без отдельного подтверждения — это может затронуть существующие данные пользователей и SQL-схему в README
- Не считай задачу завершённой без прохождения Шага 4 (проверка функциональности), включая проверку обоих интерфейсов — кнопок и команд
